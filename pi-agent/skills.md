# Skills（技能）：可复用的专业化知识包

Pi 的技能系统为 AI 编码助手提供了一套**渐进式的能力加载机制**。技能是自包含的指令包，智能体可按需加载，从而在不需要时保持上下文窗口的干净，在需要时获得深度指导。Pi 遵循 Agent Skills 开放标准，并做了一些务实的改进，确保跨工具兼容的同时适应真实世界的复杂目录结构。

## 1. 技能的位置与发现

技能可以来自多个层级，形成了**全局 → 项目 → 包 → 命令行**的加载链：

- **全局目录**：`~/.pi/agent/skills/` 和 `~/.agents/skills/`，适合个人跨项目使用的通用技能（如 PDF 处理、网页搜索）。
- **项目目录**：`.pi/skills/` 以及当前目录及祖先目录下的 `.agents/skills/`。项目技能会跟随代码仓库，实现团队共享。Pi 会自动向上查找至 git 根目录，确保团队成员无论从哪个子目录启动都能加载到相同的技能集。
- **Pi 包**：通过 `package.json` 中的 `pi.skills` 字段声明，使得技能可以作为 npm 或 git 依赖发布和安装。
- **命令行**：`--skill <path>` 可以显式加载临时技能，即使使用 `--no-skills` 也会生效，方便快速测试。

这种分层设计使得**技能的作用域自然匹配工作场景**：通用能力全局存放，业务相关的能力随项目分发，临时试验则通过 CLI 快速注入。

为了兼容其他 AI 辅助工具（如 Claude Code、OpenAI Codex），Pi 支持直接加载它们的技能目录。你只需在 `settings.json` 中配置：

```json
{
  "skills": ["~/.claude/skills", "~/.codex/skills"]
}
```

这避免了技能生态的碎片化，鼓励社区创建一套能在多个代理工具间通用的能力包。

## 2. 渐进式披露：如何工作

技能的核心设计理念是 **渐进式披露**——不在系统提示中放入大量指令，而是分两步走：

1. **启动时仅注入目录**：系统提示中只包含技能的名称和简短描述（`description`），占用极少的 token。
2. **运行时按需加载**：当模型判断当前任务与某技能描述匹配时，它会使用 `read` 工具读取完整的 `SKILL.md`。如果模型未自动触发，用户也可以通过 `/skill:name` 命令强制加载。

这种机制使得 AI 可以同时“知道”数十甚至数百个技能的存在，却只在真正需要时才展开它们的具体指令。对于长会话或频繁切换任务的场景，这显著降低了上下文污染，保持了模型的专注度。

## 3. 技能命令与用户控制

每个技能自动注册为一个 `/skill:name` 命令，用户可以主动调用：

```bash
/skill:brave-search           # 加载并执行搜索技能
/skill:pdf-tools extract      # 带参数，参数会追加到技能内容中
```

此外，通过前置元数据中的 `disable-model-invocation: true`，技能可以设为仅由用户手动触发，而不出现在系统提示的自动推荐中，适合那些副作用强或成本高的操作。

技能命令的开关可通过 `/settings` 或 `settings.json` 中的 `enableSkillCommands` 控制，给用户完全的控制权。

## 4. 技能的结构：不只是提示词

一个技能远远不止是一个 Markdown 文件，它是一个包含指令、可执行脚本、参考文档的**自包含目录**：

```
brave-search/
├── SKILL.md          # 核心：元数据 + 工作流指令
├── search.js         # 可执行脚本
├── content.js
└── references/       # 二级知识：详细 API 文档等
    └── api.md
```

- **SKILL.md**：写有前置元数据和操作步骤。它类似于一份“SOP 作业指导书”，告诉模型先做什么、再做什么、遇到问题查阅哪个文件。
- **scripts/**：可以是 Python、Shell、Node.js 等任何可执行代码。模型可以直接通过 Bash 工具调用它们，把精确的自动化操作交给脚本，自己负责规划与决策。
- **references/**：存放深层参考资料。SKILL.md 中通常只包含概要流程，具体参数和边缘情况则引导模型去读 reference 文件，形成**二级渐进加载**。

这种结构让技能成为**可执行的知识单元**，远比纯自然语言提示更可靠。

## 5. 前置元数据与验证

每个 `SKILL.md` 必须包含 YAML 前置元数据：

```yaml
---
name: brave-search
description: Web search and content extraction via Brave Search API. Use for searching documentation, facts, or any web content.
---
```

- **name**：技能的唯一标识，限小写字母、数字和连字符。Pi **不强制要求**名称与父目录一致，这在共享目录场景下（例如一个目录同时被 Claude Code 和 Pi 加载）提供了灵活性，避免了因命名规范差异导致的冲突。
- **description**：是模型决定是否自动加载技能的关键。必须具体说明“何时使用”，而不是泛泛的“帮助处理 PDF”。好的描述能大幅提升自动匹配率。

Pi 在加载时会进行验证：对于名称格式、长度等问题只发出警告，但**缺少 description 的技能会被直接跳过**，因为无法发挥渐进式披露的作用。同名技能冲突时，按照发现顺序保留第一个。

## 6. 技能与自定义工具的协同

技能（Skills）和扩展中的自定义工具（`pi.registerTool()`）是 Pi 能力体系的两个不同维度，它们经常协同工作：

- **自定义工具**赋予模型新的**操作能力**（如连接数据库、控制浏览器）。
- **技能**则提供**执行任务的流程知识**（如何分析 PDF、如何做 SEO 审计）。

一个技能可以包含脚本，而脚本内部可以使用模型本身不具备的新工具。反过来，技能也可以指导模型如何使用某个扩展提供的专用工具。两者结合，使得扩展负责“能做”，技能负责“知道怎么做”。

## 7. 技能仓库与生态

目前已有公开的技能仓库，例如 Anthropic 提供的文档处理类技能（docx、pdf、pptx、xlsx），以及 Pi 官方提供的网络搜索、浏览器自动化等技能。社区可以轻松地通过 Git 或 npm 分享技能，使得“AI 工程能力”像软件库一样被复用和迭代。

---

**总结**：Pi 的技能系统通过**渐进式披露、跨工具兼容、可执行包结构**，构建了一套可复用的 AI 能力生态。它将 AI 助手的知识加载从“全量推送”升级为“按需订阅”，既节约了上下文，又提升了任务执行的精确性和可重复性。
